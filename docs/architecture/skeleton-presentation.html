The Foundry

Skeleton pipeline — структурная схема

End-to-end каркас оркестратора: GitHub Issue → worktree → stub-план → append-line → stub-verify → реальный PR. Без LLM. Цель — проверить каркас до того, как навешивать интеллект.

реальная стадия заглушка (stub) инфраструктура будущие изменения
Слайд 2

Идея пайплайна — конечный автомат

Каждая задача — это Task, которая проходит через стадии по порядку. Состояние и current_stage пишутся в SQLite после каждого шага — можно возобновиться с падения и не потерять контекст.

Task — что проходит через пайплайн

id, issue_number, issue_title, issue_body, status, current_stage, attempts, worktree_path, branch_name, pr_url, logs_json

Почему SQLite

Персистентность между стадиями → ретраи, возобновление с падения, основа для observability. Одна таблица tasks, стdlib-only, без ORM.

Слайд 3

Entry point и оркестрация

CLI — точка входа

src/foundry/cli.py

Три команды через click: foundry run — однократный прогон, status — таблица задач из БД, reset <id> — сбросить статус для отладки.

Pipeline — FSM

src/foundry/pipeline.py

run_once() → _process_task(). После каждой стадии — upsert_task + append_log. На падении: FETCH/CONTEXT/PLAN → re-queue; IMPLEMENT/VERIFY/PR → FAILED.

Config — настройки

src/foundry/config.py

Dataclass Settings, читает .env: SOURCE_REPO, TARGET_REPO, ISSUE_LABEL, WORKTREE_ROOT, DB_PATH, GITHUB_TOKEN.

Ключевое свойство FSM: стадия сохраняется в БД до её запуска. Упали посреди — в БД видна точка падения, задача не «потерялась», лог-трейсбэк сохранён в logs_json.
Слайд 4

Стадии — кто что делает сейчас

СтадияФайлСтатусЧто делает
Fetch stages/fetch.py REAL gh issue list --label agent-task --json ... → upsert в SQLite. Возвращает pending-задачи.
Context stages/context.py STUB Возвращает {"files": []}. Интерфейс зарезервирован под реальный контекст.
Plan stages/plan.py STUB Хардкод-план: один step append_line в README.md. Формат JSON совместим с будущим LLM.
Implement stages/implement.py STUB Применяет план напрямую через FS: дописывает строку в файл worktree. Никакого Aider.
Verify stages/verify.py STUB echo ok в worktree, возвращает {"passed": true}.
PR stages/pr.py REAL git add/commit/push + gh pr create + закрытие исходного issue с комментарием.

Реальные стадии — Fetch и PR. Весь интеллект между ними — заглушки, которые будут заменены в следующих фазах.

Слайд 5

Инфраструктура — общие модули

state.py — персистентность

src/foundry/state.py

SQLite CRUD: init_db, upsert_task, get_task_by_issue, list_tasks, append_log. Одна таблица tasks, UNIQUE по (repo, issue_number).

models.py — типы

src/foundry/models.py

Stage enum (FETCH/CONTEXT/PLAN/IMPLEMENT/VERIFY/PR/DONE/FAILED), TaskStatus enum (pending/running/done/failed), dataclass Task.

shell.py — subprocess-обёртка

src/foundry/shell.py

Одна функция run(cmd, cwd, check, timeout). Используется для git, gh, verify-команд. ShellError с stdout/stderr при падении.

worktree.py — изоляция

src/foundry/worktree.py

ensure_base_repo (клонирует в worktrees/_base), create_worktree (ветка foundry/task-<id>), cleanup_worktree. Для параллельности и чистоты основного чекаута.

stages/context.py

Поиск релевантных файлов по путям из задачи, grep, семантический поиск. От простого к сложному.

stages/plan.py

Промпт: задача + контекст → структурированный JSON (какие файлы, какие изменения, какие тесты). Модель посильнее — Claude Sonnet 4.5 / GPT-5 / Gemini 2.5 Pro.

stages/implement.py

Вместо ручного FS-append: передача плана в Aider (python API) или Claude Code SDK. Патчи применяются тем же coding-агентом, не своим tool use.

pyproject.toml

Новые зависимости: anthropic (или aider-chat), openai опц., ключи в .env.

Слайд 7 · Дни 5–6

Реальная верификация + retry-loop

Замыкаем петлю качества: если verify упал — лог ошибок + текущий дифф возвращаются в implement. Лимит 3 попытки.

⇄

pytest/ruff/mypy
→

⇄ — на провале verify дифф + stderr возвращаются в implement для новой попытки

stages/verify.py

Замена echo ok на реальные pytest, ruff, mypy, билд. Таймауты, парсинг вывода, отформатированный отчёт.

pipeline.py

Ретрай-цикл в _process_task: на упавшем verify — обратно в implement с фидбэком (stderr + diff). Лимит 3, счётчик в task.attempts.

models.py

Возможно — отдельный счётчик ретраев по стадии (verify_attempts) или подтаблица попыток.

(опц.) Docker sandbox

Контейнер с окружением целевого репо, примонтированный worktree. Альтернатива — subprocess с таймаутом (проще, годится для доверенного репо).

Слайд 8 · Дни 8–14

Опциональный блок + демо

Рекомендация из драфта: Observability (Langfuse) + Agentic code review. Первое — дешёвое, второе — содержательная фича.

stages/review.py (новый)

Отдельный агент-ревьюер перед PR: читает дифф, возвращает verdict (approve / request_changes + комментарии). Замыкает петлю качества.

observability.py (новый)

Обёртки над LLM-вызовами → Langfuse (self-hosted, docker-compose). Промпты, токены, стоимость — всё видно на дашборде.

cli.py

Возможно новая команда foundry watch — poll-демон вместо однократного run. Опрос каждые 30 сек.

Web-дашборд (День 13–14)

Отдельный модуль, читает ту же SQLite. Список задач, статус, PR-ссылки, логи по стадиям. «Смотрится мощно на демо» — цитата из драфта.

День 7 — контрольная точка. К этому моменту end-to-end должен работать на реальной задаче. Если нет — всё после урезается в пользу доведения MVP.
Слайд 9

Что НЕ делаем в skeleton

Осознанные ограничения текущей фазы — чтобы не размазаться.

Нет в skeleton

  • LLM / Aider / Claude API — только заглушки
  • Реальной верификации (ruff/pytest/mypy)
  • Retry loop
  • Poll-демона — руками foundry run
  • Docker sandbox
  • Langfuse / observability
  • Веб-UI
  • Merge conflicts, code review, декомпозиции

Что требуется вручную до первого прогона

  1. brew install uv gh
  2. gh auth login (право repo)
  3. Создать sandbox-репо, лейбл agent-task, 1–2 issue
  4. cp .env.example .env + заполнить
  5. uv sync && uv run foundry run

Ожидаемый результат: в sandbox появился PR со строкой в README, issue закрыт.

Каждое «не делаем» имеет фазу, в которой появится — см. слайды 6, 7, 8. Skeleton — это каркас, а не продукт.
