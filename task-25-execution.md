## Plan

Планировщик начал работу с чтения проектных правил и поиска текущей реализации списка задач и поискового инпута. Так как `rg` в окружении агента был недоступен, разведка выполнялась через `find` и `grep` внутри текущего worktree.

В ходе разведки были выявлены основные точки изменения:

- `web/src/App.tsx` — entry point UI, где загружаются задачи, хранится статусный фильтр, считаются счетчики и рендерятся строки задач.
- `web/src/components/Topbar.tsx` — компонент, где поисковый инпут уже существовал, но был отключен через `disabled`, `aria-disabled`, `pointerEvents: "none"` и title "Поиск появится позже".
- `web/src/api.ts` — тип `UiTask`, определяющий доступные поля задачи.
- `web/src/styles.css` — существующие стили поиска, которые потенциально могли потребовать небольшой корректировки.
- `web/src/components/TaskRow.tsx` — компонент строки задачи, который менять не планировалось, но его видимые поля учитывались при выборе полей для поиска.

Выбранное решение: реализовать поиск полностью на клиенте. `App` должен хранить `searchQuery`, передавать значение и обработчик изменения в `Topbar`, а список задач фильтровать одновременно по статусу и поисковой строке. Backend API менять не требовалось, потому что `/api/tasks` уже отдавал все задачи, а существующая фильтрация по статусу также выполнялась на клиенте.

Планируемая логика поиска:

- case-insensitive поиск через `trim().toLowerCase()`;
- пустой или whitespace-only query показывает все задачи выбранного статусного фильтра;
- поиск по `issue_number`, включая варианты `42` и `#42`;
- поиск по `issue_title`, `repo`, `status`, `current_stage`, `branch_name`, `worktree_path`, `pr_url`;
- nullable-поля должны обрабатываться без ошибок;
- счетчики статусных вкладок остаются глобальными, а не пересчитываются под поисковую строку.

## Implement

Реализатор подтвердил контекст плана: поиск был отключен только в `Topbar`, а список задач уже фильтровался в `App`. После этого были внесены минимальные UI-изменения без изменения API.

Затронутые файлы:

- `web/src/App.tsx` — добавлено состояние `searchQuery`, фильтрация по статусу и поисковой строке, поиск по номеру issue, формату `#issue`, title, repo, status, stage, branch/worktree/pr URL.
- `web/src/components/Topbar.tsx` — поисковый input включен и сделан управляемым через props.

Также было обновлено пустое состояние списка:

- если задач нет вообще, остается текст `Нет задач`;
- если задачи есть, но поисковая строка и/или статусный фильтр не дают совпадений, показывается `Ничего не найдено`.

Во время проверки реализатор столкнулся с тем, что frontend-скрипты сначала не запускались: в `web` отсутствовали локальные бинарные файлы `tsc` и `eslint`, потому что зависимости не были установлены. После этого зависимости были установлены по существующему `package-lock.json` через `npm --prefix web ci`; manifest-файлы не менялись.
